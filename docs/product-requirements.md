# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**TaskCLI** - Gitと一体化した開発者向けタスク管理CLIツール

### プロダクトコンセプト
- **ターミナル完結**: GUIツールを開くことなく、ターミナルからすべてのタスク管理を完結
- **Git統合**: タスクとGitブランチを1対1で紐付け、開発フローと自然に連携
- **シンプル・高速**: 最小限のコマンドで即座に操作できる、開発者の集中を妨げない設計

### プロダクトビジョン
開発者がターミナルから離れることなく、タスク管理とコーディングを一体化できるCLIツールを提供する。
タスク開始からブランチ作成、コミット、PR作成、タスク完了までの一連の開発フローを自動化・支援し、コンテキストスイッチによる集中力の断絶をゼロにする。
JiraやAsanaのような重厚なツールとは対照的に、開発者が本当に必要な機能だけを提供し、1分で使い始められるシンプルさを維持する。

### 目的
- GUI↔ターミナル間のコンテキストスイッチを排除し、開発者の集中力を維持する
- タスクとGitブランチ・コミット・PRを自動で紐付け、手作業による管理漏れをなくす
- 個人開発者から小規模チームまでが即日導入・活用できる軽量なタスク管理基盤を提供する

## ターゲットユーザー

### プライマリーペルソナ: 田中拓也（30歳、フリーランスエンジニア）
- フリーランスで2〜4プロジェクトを並行して担当
- ターミナル中心の開発環境（tmux + Neovim）、Git操作はCLIで完結
- 現在の課題: タスクをTrelloで管理しているが、ターミナル↔ブラウザの切り替えが1日20回以上発生し、集中力が途切れる
- 期待する解決策: `task start 1` を打てばブランチも切り替わり、コミット時に自動でタスク番号が紐付く体験
- 典型的なワークフロー: `task list` で今日やることを確認 → `task start 3` でブランチ作成・切り替え → コーディング → コミット（自動でタスク番号付与） → `task done 3` で完了

### セカンダリーペルソナ: 鈴木美咲（35歳、スタートアップのテックリード）
- 3〜5人の小規模チームを率いるリードエンジニア
- 現在の課題: Jira/Asanaは機能過多で導入コストが高く、チームの現状把握に時間がかかる
- 期待する解決策: `task list --team` で朝会前に30秒でチームの進捗を確認できる
- 典型的なワークフロー: 朝 `task list --team` で進捗確認 → 朝会 → 自分のタスクを `task start` で開始 → 夕方 `task done` で完了・PR作成

## 機能要件

### コア機能（MVP / P0）

#### 1. タスクの基本操作（CRUD）

**ユーザーストーリー**:
開発者として、ターミナルから離れずにタスクを管理するために、タスクの追加・一覧表示・詳細確認・削除ができる機能が欲しい

**受け入れ条件**:
- [ ] `task add "タスク名"` でタスクを作成でき、一意のIDが自動採番される
- [ ] `task list` で全タスクをID・ステータス・タイトル・関連ブランチを含む一覧で表示できる
- [ ] `task show <id>` でタスクの全詳細（ID・タイトル・ステータス・作成日時・関連ブランチ・説明）を確認できる
- [ ] `task delete <id>` でタスクを削除でき、削除前に確認プロンプトが表示される
- [ ] コマンド実行時間が100ms以内（平均的なPC環境）

**優先度**: P0（必須）

---

#### 2. タスクのステータス管理

**ユーザーストーリー**:
開発者として、作業の進捗を把握するために、タスクのステータスを管理したい

**受け入れ条件**:
- [ ] ステータスは `open`（新規）、`in_progress`（作業中）、`completed`（完了）、`archived`（アーカイブ）の4種類が存在する
- [ ] `task start <id>` でステータスを `in_progress` に変更できる
- [ ] `task done <id>` でステータスを `completed` に変更できる
- [ ] `task list` でステータスごとに色分けして表示される（open: 白、in_progress: 黄、completed: 緑）
- [ ] `task list --status <status>` で特定ステータスのタスクのみ絞り込める

**優先度**: P0（必須）

---

#### 3. GitブランチとのP0連携

**ユーザーストーリー**:
開発者として、タスクとブランチを自動で紐付けてもらうために、タスク開始時に自動でブランチを作成・切り替えする機能が欲しい

**受け入れ条件**:
- [ ] `task start <id>` 実行時に `feature/task-<id>-<タイトルのslug>` 形式のブランチが自動で作成される
- [ ] ブランチ作成後、自動でそのブランチに切り替わる
- [ ] すでに同名ブランチが存在する場合は既存ブランチに切り替わり、エラーにならない
- [ ] Gitリポジトリ外で実行した場合、Gitなしモード（ローカルタスクのみ）で動作するか、わかりやすいエラーメッセージを表示する
- [ ] `task list` のBranch列に紐付いているブランチ名が表示される

**優先度**: P0（必須）

---

#### 4. JSONによるデータ永続化

**ユーザーストーリー**:
開発者として、データが失われる心配なくタスクを管理するために、タスクデータが確実に保存・読み込まれる必要がある

**受け入れ条件**:
- [ ] タスクデータは `.task/tasks.json` にJSON形式で保存される
- [ ] プロセス終了・再起動後もデータが保持される
- [ ] JSONファイルはGitで管理可能（チーム間での共有が可能）
- [ ] JSONファイルが破損した場合、わかりやすいエラーメッセージとリカバリ手順を表示する
- [ ] タスク操作前に自動でバックアップ（`.task/tasks.json.bak`）を作成する

**優先度**: P0（必須）

---

### 重要機能（P1）

#### 5. GitHub Issuesとの同期

**ユーザーストーリー**:
開発者として、既存のGitHub Issuesと重複管理を避けるために、ローカルタスクとGitHub Issuesを同期したい

**受け入れ条件**:
- [ ] `task sync` でローカルタスクとGitHub Issuesを双方向同期できる
- [ ] `task import --github` でGitHub IssuesをローカルタスクにインポートできるI
- [ ] Personal Access Tokenを `.task/config.json` に設定して認証できる
- [ ] 同期時に競合が発生した場合、どちらを優先するか確認プロンプトが表示される

**優先度**: P1（重要）

---

#### 6. タスク完了時の自動処理

**ユーザーストーリー**:
開発者として、タスク完了後の手作業を減らすために、`task done` 実行時にPR作成などを自動化したい

**受け入れ条件**:
- [ ] `task done <id> --pr` でGitHub PRを自動作成できる
- [ ] PR作成時、タイトルにタスク番号とタイトルが自動設定される
- [ ] `--no-merge` オプションなしの場合はmainブランチへのマージを提案する
- [ ] 各ステップの実行前に確認プロンプトを表示し、スキップ可能にする

**優先度**: P1（重要）

---

#### 7. タスクの検索・絞り込み

**ユーザーストーリー**:
開発者として、多数のタスクの中から素早く目的のタスクを見つけるために、検索・絞り込み機能が欲しい

**受け入れ条件**:
- [ ] `task search "<キーワード>"` でタイトル・説明を対象にキーワード検索できる
- [ ] `task list --status <status>` でステータスで絞り込める
- [ ] `task list --sort priority` で優先度順にソートできる
- [ ] 検索結果は100ms以内に表示される（1000件のデータで）

**優先度**: P1（重要）

---

#### 8. 優先度と期限の管理

**ユーザーストーリー**:
開発者として、重要なタスクを見落とさないために、優先度と期限を設定・管理したい

**受け入れ条件**:
- [ ] `task add "タスク名" --priority <high|medium|low>` で優先度を設定できる
- [ ] `task add "タスク名" --due <YYYY-MM-DD>` で期限を設定できる
- [ ] `task list` で期限切れのタスクが赤色でハイライトされる
- [ ] `task list --sort priority` で優先度の高い順に表示される

**優先度**: P1（重要）

---

### 将来的な機能（Post-MVP / P2）

#### 9. チーム機能

`task list --team` でチームメンバー全員のタスク一覧を表示し、`task assign <id> @username` で担当者を変更できる。

**優先度**: P2（できれば）

---

#### 10. 作業時間の記録

`task start <id>` 時に計測を開始し、`task done <id>` 時に経過時間を記録・表示する。

**優先度**: P2（できれば）

---

#### 11. タスクテンプレート

`task template create <name>` でテンプレートを定義し、`task add --template <name> "タイトル"` で定型タスクを素早く作成できる。

**優先度**: P2（できれば）

---

#### 12. カレンダービュー

`task calendar --week` / `--month` で期限ベースのカレンダー表示ができる。

**優先度**: P2（できれば）

---

### CLIインターフェース

```bash
# タスクの基本操作
task add "ユーザー認証機能の実装"
task add "緊急バグ修正" --priority high --due 2025-02-01
task list
task list --status in_progress
task list --sort priority
task show 1
task done 1
task delete 1

# Git連携
task start 1
# → feature/task-1-user-authentication ブランチを自動作成・切り替え

# 検索
task search "認証"

# GitHub連携（P1）
task sync
task import --github
task done 1 --pr

# チーム機能（P2）
task list --team
task assign 1 @alice
```

## 非機能要件

### パフォーマンス
- コマンド実行時間: 100ms以内（平均的なPC環境、GitHub API呼び出しを含まない操作）
- タスク一覧表示: 1000件まで1秒以内
- 起動時間: 500ms以内（Node.js起動込み）

### ユーザービリティ
- 新規ユーザーが `task --help` のみを見て5分以内に基本操作（add・list・start・done）を習得できる
- すべてのコマンドに `--help` オプションが存在し、使用例が含まれる
- エラーメッセージには問題の原因と解決策を日本語・英語で明示する

### 信頼性
- タスク操作はアトミックに実行され、途中失敗時にデータが破損しない
- 操作前の自動バックアップにより、誤操作後もデータを復元できる
- 危険な操作（delete、強制上書き）には必ず確認プロンプトを表示する

### セキュリティ
- GitHub Personal Access Tokenは `.task/config.json` に保存し、`.gitignore` への追加を自動提案する
- 外部APIへのHTTPS通信のみを許可する
- コマンドインジェクション等のセキュリティリスクをもたらす入力を適切にサニタイズする

### スケーラビリティ
- JSONストレージで最大10,000件のタスクを管理できる（それ以上はSQLite移行を推奨）
- 将来のSQLite移行のために、ストレージ層を抽象化して実装する

### 互換性
- macOS、Linux（Ubuntu 20.04以上）、Windows（Git Bash環境）で動作する
- Node.js v18以上で動作する
- GitLabやBitbucketのリポジトリでもGit操作の基本機能は動作する（GitHub固有機能はGitHubのみ）

## スコープ外

明示的にスコープ外とする項目:

- **GUIインターフェース**: ターミナルへの集中という設計思想に反するため、MVP・P1には含めない
- **クラウド同期サービス**: Git経由での共有で十分。専用クラウドバックエンドの開発はP2以降
- **GitLab/Bitbucket固有のAPI連携**: GitHub Issuesとの連携を優先し、他のプロバイダーはP2以降
- **モバイルアプリ**: 開発者のターミナル環境に特化するため、モバイルは対象外
- **課金・サブスクリプション機能**: 初期リリースはOSSとして完全無料で提供
