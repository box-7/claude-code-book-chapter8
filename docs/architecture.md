## アーキテクチャパターン

### レイヤードアーキテクチャ

```
┌─────────────────────────────────────────┐
│   CLIレイヤー (src/cli/)                 │  ← ユーザー入力の受付・パース・表示
├─────────────────────────────────────────┤
│   サービスレイヤー (src/services/)        │  ← ビジネスロジック
├─────────────────────────────────────────┤
│   ストレージ抽象化レイヤー (src/storage/) │  ← データアクセス抽象化
├─────────────────────────────────────────┤
│   データレイヤー (.task/)               │  ← JSONファイル / 将来: SQLite
└─────────────────────────────────────────┘
```

#### CLIレイヤー (`src/cli/`)
- **責務**: ユーザー入力の受付・バリデーション、結果の整形・表示
- **許可される操作**: サービスレイヤーの呼び出し、chalk/cli-table3の使用
- **禁止される操作**: データレイヤーへの直接アクセス、ビジネスロジックの実装

#### サービスレイヤー (`src/services/`)
- **責務**: タスク管理・Git操作・GitHub連携のビジネスロジック
- **許可される操作**: ストレージ抽象化レイヤーの呼び出し、simple-git/Octokitの使用
- **禁止される操作**: CLIレイヤーへの依存、UIの表示ロジック

#### ストレージ抽象化レイヤー (`src/storage/`)
- **責務**: データアクセスの抽象化、バックアップ管理
- **許可される操作**: ファイルシステムへのアクセス
- **禁止される操作**: ビジネスロジックの実装

**依存関係の方向**:
```
CLI → Services → Storage → ファイルシステム
              → GitService (simple-git)
              → GitHubService (Octokit)
```

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| タスクデータ | ローカルファイル `.task/tasks.json` | JSON | シンプル・Gitで共有可能・外部依存なし |
| 設定データ | ローカルファイル `.task/config.json` | JSON | 軽量・人間が読みやすい |
| バックアップ | ローカルファイル `.task/tasks.json.bak` | JSON | 直前の状態を保持、誤操作からの復元用 |

### バックアップ戦略

- **タイミング**: 書き込み操作（add/update/delete）の直前に自動バックアップ
- **保存先**: `.task/tasks.json.bak`（世代管理は1世代）
- **復元方法**: `task restore` コマンド、または手動で `.bak` ファイルをリネーム

### 将来のSQLite移行

10,000件超のタスクを管理する場合、ストレージ抽象化レイヤーを介してSQLiteへ移行する。
`FileStorage` と同一インターフェース (`IStorage`) を実装した `SqliteStorage` クラスを追加するだけで移行可能な設計とする。

```typescript
interface IStorage {
  load(): StorageData;
  save(data: StorageData): void;
  backup(): void;
  exists(): boolean;
  initialize(): void;
}
```

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| `task list`（100件） | 100ms以内 | CPU Core i5相当、メモリ8GB、SSD |
| `task list`（1000件） | 1000ms以内 | 同上 |
| `task add` / `task done` | 100ms以内（Git操作なし） | 同上 |
| `task start`（ブランチ作成含む） | 500ms以内 | 同上 |
| Node.js起動時間 | 500ms以内 | 同上 |

### リソース使用量

| リソース | 上限 | 理由 |
|---------|------|------|
| メモリ | 50MB | 全タスクをメモリに展開しても問題ない上限。1万件×1KB=10MB程度 |
| ディスク（データ） | 100MB | `.task/tasks.json` の上限目安 |

## セキュリティアーキテクチャ

### データ保護

- **アクセス制御**: `.task/config.json`（GitHubトークン含む）のファイルパーミッションを `600`（所有者のみ読み書き）に自動設定
- **機密情報管理**: GitHubトークンは `.task/config.json` のみに保存。`task init` 時に `.gitignore` への追加を自動提案
- **暗号化**: MVPフェーズでは非暗号化。将来的にOS標準のキーチェーン（macOS Keychain / Linux Secret Service）への保存を検討

### 入力検証

- **バリデーション対象**: タイトル（1〜200文字）、日付形式（YYYY-MM-DD）、priority値（high/medium/low）、タスクID（先頭6文字以上の一致）
- **サニタイゼーション**: ブランチ名生成時はタイトルを英数字・ハイフンのみのslugに正規化（日本語→URLエンコード後に短縮）
- **コマンドインジェクション対策**: `simple-git` のAPIのみを使用し、シェル文字列補間によるGitコマンド構築を禁止

## スケーラビリティ設計

### データ増加への対応

- **想定データ量**: 通常運用で〜1,000件、上限目安10,000件
- **パフォーマンス劣化対策**: 10,000件を超えた場合は `task archive --completed --before <date>` でアーカイブを推奨
- **アーカイブ戦略**: 完了タスクを `.task/archive-YYYY.json` に移動し、アクティブな `tasks.json` を軽量に保つ

### 機能拡張性

- **ストレージ抽象化**: `IStorage` インターフェースにより、JSONからSQLiteへの移行が容易
- **コマンド追加**: Commander.jsのサブコマンド構造により、新コマンドをファイル単位で追加可能
- **設定のカスタマイズ**: `.task/config.json` で `defaultBranch`、`autoBackup` 等を設定可能

## テスト戦略

### ユニットテスト
- **フレームワーク**: Vitest
- **対象**: `TaskManager`、`GitService.generateBranchName`、`FileStorage`、入力バリデーション関数
- **カバレッジ目標**: 80%以上（サービスレイヤーは90%以上）
- **モック方針**: `simple-git` と `fs` をモック化して高速に実行

### 統合テスト
- **方法**: 実際のファイルシステム（一時ディレクトリ）を使用したテスト
- **対象**: `task add → task start → task done` の一連フロー、`FileStorage` のCRUD操作

### E2Eテスト
- **ツール**: Node.js `child_process` でCLIコマンドを実際に実行
- **シナリオ**: Gitリポジトリあり・なし両環境での基本操作
- **CI**: GitHub ActionsでmacOS / Ubuntu / Windows（Git Bash）の3環境で実行

## 技術的制約

### 環境要件
- **OS**: macOS 12以上、Ubuntu 20.04以上、Windows 10以上（Git Bash環境）
- **Node.js**: v18以上（`fs/promises`・`crypto.randomUUID` の利用）
- **Git**: v2.20以上（ブランチ操作に使用するオプションの互換性）
- **最小メモリ**: 256MB
- **必要ディスク容量**: 50MB（node_modules含む）

### パフォーマンス制約
- Node.jsの起動オーバーヘッドにより、最速でも50〜100ms程度の遅延が発生する（構造上の制約）
- JSON全件読み込みのため、10,000件超ではメモリ・パース時間が線形増加する

### セキュリティ制約
- GitHub Personal Access Tokenをコード内にハードコードすることを禁止
- `.task/config.json` を `.gitignore` に追加せずにコミットすることを禁止（`task init` で自動チェック）
