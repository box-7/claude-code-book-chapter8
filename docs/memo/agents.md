# .claude/agents/ メモ

## エージェントとは何か

`.claude/agents/` に置いたMarkdownファイルは、特定の専門作業を担う「サブエージェント」として機能する。

スキル（`.claude/skills/`）との違いは、エージェントはメインのClaude Codeとは**別のClaude**として独立して動く点。
メインの会話を邪魔せず、専門的な分析や検証を並行して任せられる。
レビューや検証のように「客観的な第三者視点が欲しい」作業に向いている。

---

## doc-reviewer

**ドキュメントの品質をレビューするエージェント。**

PRD・機能設計書・アーキテクチャ設計書などのドキュメントを読んで、「このドキュメント、大丈夫か？」を5つの観点でチェックして改善提案を出す。
`/review-docs` コマンドから呼び出される。

### 5つのレビュー観点

1. **完全性** — 必要なセクションや情報が抜けていないか
2. **明確性** — 誰が読んでも同じ意味に解釈できるか（曖昧な表現がないか）
3. **一貫性** — 他のドキュメントと矛盾していないか、用語の表記が統一されているか
4. **実装可能性** — 開発者がこれを読んで実際に実装できるだけの情報があるか
5. **測定可能性** — 「高速であること」ではなく「100ms以内」のように数値で判断できるか

### レビュー結果の形式

各観点を ✅/⚠️/❌ と1〜5のスコアで評価し、以下の3段階で改善提案を出す:
- **[必須]** — すぐ直さないといけない重大な問題
- **[推奨]** — できれば直したほうがいい点
- **[提案]** — 余裕があれば検討してほしい改善案

良い点も必ず3つ以上挙げて、批判一辺倒にならないようにしている。

### ドキュメント種別ごとの追加チェック

ドキュメントの種類によって追加のチェック項目がある。
たとえばPRDならKPIが定義されているか、機能設計書ならエラーハンドリングが考慮されているかなど、それぞれの種類に固有の観点でも見る。

## implementation-validator

**実装コードがスペック通りに作られているかを検証するエージェント。**

コードを書いた後に「ちゃんとPRDや設計書の通りに実装できているか？」「品質に問題はないか？」をチェックする。
コードを書いた本人が見落としがちな問題を客観的に発見するのが目的。

### 5つの検証観点

1. **スペック準拠** — PRD・機能設計書・アーキテクチャ設計書に書いてある通りに実装されているか
2. **コード品質** — 命名規則・単一責務の原則・重複コードなど、コーディング規約を守れているか
3. **テストカバレッジ** — テストが十分に書かれているか（目安: 80%以上で合格）
4. **セキュリティ** — パスワードや APIキーをコードに直書きしていないか、入力値の検証があるかなど
5. **パフォーマンス** — 無駄な処理やメモリリークの可能性がないか

### 検証の流れ

1. `docs/` 以下のスペックドキュメントを読んで「本来どう動くべきか」を理解する
2. 実装コードを読んで構造を把握する
3. 5つの観点でチェックする
4. `npm run lint`・`npm run typecheck`・`npm test` などのコマンドを実際に実行して結果を確認する
5. 問題があれば「どのファイルの何行目が・なぜ問題で・どう直すか」を具体的に報告する

### doc-reviewer との違い

| | doc-reviewer | implementation-validator |
|---|---|---|
| 対象 | Markdownドキュメント | TypeScript実装コード |
| 目的 | ドキュメントの品質向上 | コードのスペック準拠・品質確認 |
| タイミング | ドキュメント作成後 | コード実装後 |
